    fix(auth): exclude /admin/reset-password from middleware's auth gate
    
    Supabase's password-recovery link lands on /admin/reset-password with
    the session token in the URL hash fragment (#access_token=...&type=
    recovery), which browsers never send to the server. Middleware's
    getUser() check saw no session on that first request and redirected
    straight to /admin/login before the page's client-side code ever got
    to read the hash and exchange it for a session — silently breaking the
    password-reset flow end to end, even though the "Forgot password?"
    link and email itself were already fixed correctly.
    
    Verified: unauthenticated /admin/reset-password now renders the form
    instead of redirecting; /admin still correctly gates to login.
